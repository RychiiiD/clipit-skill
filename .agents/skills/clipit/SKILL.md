说一句话就开始：

```text
帮我粗剪这个视频，主题是远程求职技巧
```

## 触发关键词

当你需要粗剪、剪辑、裁剪视频、切视频、剪掉废话、去掉跑题、去掉停顿、视频切片、切片导出时，使用此技能。

## 总体原则

1. **纯自然语言交互** — 用户不敲任何命令，只说一句话
2. **clipit core 只做本地音视频处理** — 语义分析由 LLM 自身完成
3. **预览确认** — 执行裁剪前必须展示给用户确认
4. **跨平台设计** — 不假设任何特定 Agent 平台或操作系统

## 安装流程

用户说类似"帮我装一下 clipit"时触发：

```text
1. 检测 Python 是否可用 → 如不可用则报错引导
2. pip install clipit（或本地 pip install -e .）
3. 调 clipit check 检查环境
4. 检查输出中的 clipit_cmd.available 和 warnings 字段：
   - available=false → 按 suggestion 将 Python Scripts 目录加入 PATH
   - 有 ffmpeg warning → 调 clipit install
5. 再次调 clipit check 确认所有项正常
6. 告知用户安装完成，可开始使用
```

**不配置任何 API Key** — 语义分析用 LLM 自带能力。

## 文件目录约定

所有过程文件和输出统一放在当前工作目录下的 `data/` 中：

```
data/
├── process-data/          ← 中间文件
│   ├── transcript.json         ← Step 2 转字幕结果（原始）
│   ├── transcript_clean.json   ← Step 2.5 清洗后字幕
│   └── decisions.json          ← Step 3 AI 裁剪决策
└── output/
    └── <原文件名>_output.mp4  ← Step 5 最终成品
```

目录自动创建，不需用户手动操作。

## 粗剪管线

### Step 1 — 意图提取

从用户 NL 中提取：
- `video_path` — 视频文件路径（用户拖入或输入）
- `topic` — 主题描述
- `clip_intensity` — 剪辑强度（默认 medium，用户可说"剪严格一点"调整）

未提供视频路径 → 请用户拖入或输入路径。
未提供主题 → 请用户描述视频主题。

### Step 2 — 转字幕

```bash
mkdir -p data/process-data
clipit transcribe <video_path> --model small -o data/process-data/transcript.json
```

读取 `data/process-data/transcript.json` 中的 `segments` 列表，向用户展示概览：
- 总时长
- 总句数
- 检测到的语言

### Step 2.5 — 字幕清洗

```bash
clipit clean data/process-data/transcript.json -o data/process-data/transcript_clean.json
```

对 `data/process-data/transcript.json` 中的每段字幕文本执行清洗，结果保存到 `data/process-data/transcript_clean.json`（格式与原文件一致，仅 `text` 字段变化）。

清洗规则：

| 模式 | 例子 | 清洗后 |
|------|------|--------|
| 重复词 | "就，就是"、"他，他们" | "就是"、"他们" |
| 填充词 | "嗯... 就是... 这个..." | 去掉 |
| 句子开头卡顿 | "所以... 所以我认为" | "所以我认为" |
| 相邻段首尾重复 | 段A以"所以"结尾，段B以"所以"开头 | 段结尾不变，段B开头去重 |

清洗**不修改时间戳**，只改 `text` 字段。

### Step 3 — 语义分析（口播精编）

用 `data/process-data/transcript_clean.json`（已清洗过的字幕）中的整段字幕文本构造 Prompt，让 LLM 从"写一篇完整口播稿"的角度做精编：

```text
你是一个视频口播精编助手。给定一段视频字幕，从写一篇**完整的口播稿**的角度，筛选出最能连贯表达主题的核心内容。你的输出是一篇自洽的短视频脚本，不是原始视频的子集。

## 结构性要求（必须同时满足）

### 1. 完整论证链
保留的内容必须构成完整的论述结构。一篇合格的口播需包含：
- **开头钩子（前6秒）**: 视频第一段保留内容必须是直接点明主题的陈述句，不能是自我介绍、背景铺垫或互动寒暄。未点明主题前不保留任何内容。
- **论点→论据→案例→方法论→结论**: 每个论证单元必须完整。不能只保留"论点+结论"而切掉中间的分析推导过程。
- **禁止碎片化输出**: 按"段落"粒度而非"句子"粒度做 keep/cut。同一论据/案例/推导过程中连续的句子必须合并为一个 keep 段，不能切成多个小段。keep 段数量控制在 3-8 段，每个 keep 段不少于 5s（极短金句除外）。如果输出超过 10 段说明你在逐句分类而非成文精编，必须合并。
- **结论性收尾**: 视频最后一段保留内容必须是完整的结论句，不能是反问、提问或悬空句。如果原始视频结尾是提问，往前找到最近的结论句作为收尾。

### 2. 完整案例规则
当保留任何一个案例/故事时，必须保留它的**完整叙述弧**：案例引入 → 细节展开 → 分析解读 → 结论提炼。不能只保留引入和结论而切掉中间的推导，也不能只保留案例的某一半。

**特别注意：案例开场钩子紧挨跑题内容时，不要误伤。** 案例引入句（如"我来给你们讲个真实案例"）即使后面紧跟着一段与主题无关的观众互怼/闲聊，也要保留引入句。正确做法是：保留案例开场钩子 → 裁掉跑题段 → 在后续恢复案例的展开和结论。不能因为案例的展开段被跑题打断，就连案例开场一起抛弃。

### 3. 禁止保留的内容
以下内容类型无论是否与主题沾边，一律裁掉：
- **弹幕/观众互动**: "知道的扣1不知道的扣2"、"你们觉得呢"、"把XX打在公屏上"
- **抬杠/对线**: 主播与观众争辩的内容（"你来我这儿讲"、"就显着你了"）
- **自证/卖惨**: "我98年的"、"我28岁有三家公司"等自我介绍/credential展示
- **直播间尬聊**: "今天怎么这么漂亮"、"我这么智慧的发言"等与主题无关的闲谈

### 4. 多主题处理
同一视频剪多个主题时，每个主题的粗剪结果必须是**独立完整的口播稿**。不假设观众看过其他主题。每个主题都应包含完整的开头钩子、论证结构和结论收尾。

**严格禁止跨主题保留。** 其他主题的内容即使与本主题在概念上有交叉（如"第一性原理"和"生产端思维"都涉及拆解问题），也不能保留。判断标准：这段内容直接服务于指定主题的论证链吗？如果是其他主题的论据或案例，无论多精彩都裁掉。

### 5. 时间顺序
splice 默认按原始时间顺序拼接。如果主题句出现在视频中段，前序内容必须在逻辑上为引出主题服务（如设问→冲突→解答的结构）。不要保留与主题无关的前序内容。

**可选：段落重排序** — 如果你认为某段金句/主题句应该在口播开头而非它在视频中的原始位置，可在 keep 段中添加 `"order": int`（从1开始）。输出时 splice 会按 order 排序而非时间顺序。两个 keep 段使用相同 order 值时按时间顺序拼接。

## 可选保留（提升内容质量）

以下类型如有空间可考虑保留，但不是必须：
- **类比/比喻**: 帮助理解的类比（如"马车 vs 汽车"的历史对比）
- **递进案例**: 从第一个案例自然引申到第二个案例（如从"造火箭"到"造新能源车也是同样思路"）
- **insight总结**: "所以你会发现真正赚到钱的人……"这类升华句

## 「内容重复」vs 递进修辞 判断标准

标记为"内容重复"时，严格区分：

- **机械重复 → "内容重复"**：同一句话几乎原样连续说两遍或多遍（如"这个很重要"紧接着"这个很重要"）。这种情况保留一句即可。
- **递进修辞 → 保留**：同一概念换说法递进情绪（如"这个方法效率很高"紧接着"这完全改变了行业规则"）。口播中这是递进，不是重复，后一句话比前一句在语义/情绪上更强或更深。

判断方法：如果后一句和前一句意思完全一样且没有增强 → 重复；如果后一句比前一句语义更强、情绪更深或换了角度深化 → 递进。

## Reason 取值规则

reason 字段必须且仅限以下 5 种之一：

- "内容保留" — 核心内容或必要铺垫
- "跑题" — 与主题完全无关、或禁止保留类型（弹幕互动、抬杠、自证等）
- "例子过长" — 举例篇幅远超论证需要
- "内容重复" — 已表达过的观点反复说
- "空白停顿" — 嗯、啊、沉默等无效内容

## 输出前自检清单

在返回 JSON 之前，逐项检查：
1. [ ] 输出的第一段保留内容是否直接点明主题？（不是寒暄/自我介绍/背景）
2. [ ] 输出的最后一段是否为完整的结论句？（不是反问/提问/悬空）
3. [ ] 是否有案例被截断？（案例必须有引入+展开+分析+结论，不能只保留片段）
4. [ ] 是否有弹幕互动/抬杠/自证内容混入？
5. [ ] 保留段连起来能否读成一篇通顺的口播？（在脑中顺序朗读一遍）
6. [ ] 是否形成了"论点→论据→案例→方法论（可选）→结论"的完整结构？
7. [ ] 案例开场是否因紧挨跑题内容而被连带切掉了？案例引入句应独立保留
8. [ ] keep 段是否超过 10 段？如果超过，说明你在逐句分类，需要合并为段落级 keep
9. [ ] 是否有其他主题的内容混入？只保留当前主题的完整口播，其他主题内容一律裁掉

用户主题：{topic}
剪辑强度：{intensity}

字幕内容（含时间戳，已清洗过重复/填充词）：
{segments}

按此格式返回 JSON：
[{"start": float, "end": float, "action": "keep"|"cut", "reason": "内容保留"|"跑题"|"例子过长"|"内容重复"|"空白停顿"}]

可选字段（仅对 keep 段生效）：
- "order": int — 输出顺序号（从1开始），用于非时间顺序排列段落。当你想让金句/主题句出现在开头、而背景内容在其后时使用。不填则按原始时间顺序输出。
```

将生成的决策 JSON 保存到 `data/process-data/decisions.json`。

### Step 3.5 — 硬规则验证

LLM 不可靠，clipit 用代码强制规则修正 LLM 决策：

```bash
clipit validate data/process-data/decisions.json -i <intensity>
```

强制规则（**AI 不可覆盖**）：

| 规则 | 说明 |
|------|------|
| R1 开篇定题 | 开篇引入段强制保留（仅保护2-20s的合理开篇，超过20s的长段前摇不保护）；首段keep不超3s时连保第二段 |
| R2 短段合并 | keep 片段 < min_keep 自动合并/扩展 |
| R3 裁剪上限 | 总裁剪比例不超过 intensity 阈值，超限时恢复边缘段落 |
| R4 同动作去重 | 相邻 keep/keep 或 cut/cut 自动合并 |
| R5 Reason 校验 | reason 不在 5 种标准值内时自动标准化（保障 R3 优先级排序有效） |
| R6 流畅度防护 | keep 段过多或平均过短时裁掉孤立岛（<3s 两头切），桥接短间隙（<max_gap） |

向用户展示验证变更摘要：

```text
┌─────────────────────────────────────┐
│ 验证修正（代码强制）                  │
│ {n} 处变更                            │
│ • R1 开篇定题: [{t}-{t}] 强制保留    │
│ • R2 短段合并: [{t}-{t}] → 下一段     │
│ • R3 裁剪超限: 恢复 {n} 段            │
│ • R5 reason校验: {n} 处              │
│ • R6 流畅度修正: {n} 处              │
│                                      │
│ 保留: {n}s ({n}%)  裁剪: {n}s        │
└─────────────────────────────────────┘
```

### Step 4 — 用户确认

如果 LLM 生成的 decisions 中有 `order` 字段（段落重排序），在展示预览时额外询问用户：

### Step 5 — 执行拼接

```bash
mkdir -p data/output
# 默认：按原始时间顺序拼接（默认）
clipit splice <video_path> -d data/process-data/decisions.json -o data/output/<原文件名>_output.mp4

# 如需段落重排：加上 --reorder（按 decisions 中 order 字段排序）
clipit splice <video_path> -d data/process-data/decisions.json --reorder -o data/output/<原文件名>_output.mp4
```

## 参数调整（纯 NL 映射）

| 用户说 | 对应操作 |
|--------|---------|
| "剪严格一点" | intensity → strict，重新 Step 3 |
| "宽松点，别剪太多" | intensity → loose，重新 Step 3 |
| "保留片段别短于 3 秒" | min_keep → 3.0，重新 Step 3 |
| "片段前后留半秒过渡" | transition → 0.5，重新 Step 5 |

### Intensity 选型指引

根据目标保留比例选择 intensity：
- **loose**（max_cut_pct=50）: 保留 > 50%，宽松裁剪，仅去明显废话
- **medium**（max_cut_pct=70）: 保留 30-50%，一般粗剪（默认）
- **strict**（max_cut_pct=85）: 保留 15-30%，精编式裁剪
- **aggressive**（max_cut_pct=92）: 保留 8-15%，极度精简，仅保留最核心的口播内容

经验：当从长视频（>20min）中提取极少量精华内容时（如21分钟视频保留2-3分钟），使用 aggressive 避免 R3 恢复不需要的边缘内容。

## CLI 参考（Agent 内部调用，不对用户展示）

```bash
clipit check                    # 环境检查
clipit install                  # 自动安装 ffmpeg
clipit transcribe <video>       # 转字幕，输出 JSON
clipit clean <transcript.json>  # 清洗字幕（去填充词/重复/卡顿）
clipit splice <video> -d decisions.json  # 按决策拼接
```

> **编码保护**：`transcribe` 使用 `-o` 写文件（绕过 stdout 编码问题）；所有 JSON 输出都强制 UTF-8。如遇编码问题可在命令前加 `PYTHONIOENCODING=utf-8`。<br>
> 所有命令输出 JSON，无需额外解析。
